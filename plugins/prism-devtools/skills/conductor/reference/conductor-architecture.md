# Conductor Architecture

## Overview

Conductor selects the best prompt variant for each agent/step
combination, injects system context from Brain, and learns from
execution outcomes. It wraps `build_agent_instruction()` without
changing the stop hook's flow.

## How It Fits in the Workflow

```
prism-loop fires step
       |
       v
 CONDUCTOR
 1. Estimate difficulty (token history, AC count)
 2. Pick best prompt (epsilon-greedy from scores.db)
 3. Query Brain for system context (per-AC search)
 4. Inject role card + system context into instruction
 5. Auto-reindex if stale
       |
       v
 AGENT executes with optimized prompt + context
       |
       v
 STOP HOOK records outcome -> scores.db + outcomes.jsonl
              triggers incremental_reindex()
```

## Integration Point

Conductor replaces ONLY the role card selection in
`build_agent_instruction()`. All other parts (step body, inline
rules, stop directive) are unchanged.

```python
# Fallback is always available — zero regression if Conductor fails
try:
    from conductor_engine import Conductor
    conductor = Conductor()
    prompt, prompt_id, context_tokens = conductor.build_prompt(
        agent_id, step_id, story_file=story_file, task_context=task_context
    )
except (ImportError, Exception):
    prompt = None  # Falls back to static ROLE_CARDS
```

## Prompt Variant Storage

### Default variants (shipped in plugin, read-only)
```
plugins/prism-devtools/prompts/
├── sm/{default,lean,rich}.md
├── qa/{default,lean,rich}.md
├── dev/{default,lean,rich}.md
└── architect/{default,lean,rich}.md
```

### Learned variants (per-project, in scores.db)
Stored in `scores.db:prompt_variants` table. Generated by
`/conductor generate` after sufficient outcome data.

### Resolution order
1. Check `scores.db:prompt_variants` for project-specific variant
2. Fall back to plugin's `prompts/{persona}/{variant}.md`
3. Fall back to static ROLE_CARDS in prism_loop_context.py

## Scoring (PSP-Based)

Each outcome is scored using PSP metrics from quality gate artifacts:

```
score = (
    0.40 * gate_passed          # Quality gate pass/fail
  + 0.20 * coverage_pct/100     # Test coverage
  + 0.20 * traceability_pct/100 # AC traceability
  + 0.10 * probe_accuracy       # Estimation accuracy
  - 0.05 * (tokens_used / avg_tokens)  # Token efficiency
  - 0.05 * retries              # Retry penalty
)
```

## Exploration vs Exploitation

Conductor uses epsilon-greedy strategy:

- **epsilon_start**: 0.3 (30% exploration initially)
- **epsilon_min**: 0.05 (5% floor — always some exploration)
- **decay_per**: 50 outcomes to decay by half
- **min_outcomes**: 5 before switching from pure exploration

With probability epsilon: pick random variant (explore).
With probability 1-epsilon: pick highest-scoring variant (exploit).

## Variant Retirement

Poor-performing variants are retired to prevent score pollution:

- **min_runs**: 10 (minimum runs before retirement eligible)
- **threshold**: -1.0 (score below this is "poor")
- **consecutive**: 5 (consecutive poor runs triggers retirement)

Retired variants move to `scores.db:retired_variants` and are
excluded from selection. They can be unretired manually.

## Variant Generation

After 20+ outcomes, `/conductor generate <persona> <step>`:
1. Fetches top 5 and bottom 5 outcomes for the persona/step
2. Analyzes what differs between high-scoring and low-scoring runs
3. Generates a new variant hypothesis
4. Inserts into `scores.db:prompt_variants` with score=0 (untested)
5. New variant enters exploration pool

## System Context Injection

For each step, Conductor queries Brain to build a context block:

- **Pre-story steps** (steps 1-2): uses task_context from JIRA/PRD/text
- **Post-story steps** (steps 3+): uses story ACs as search queries

Brain returns ranked chunks; Conductor formats them as a
`## System Context` section injected into the agent instruction.
Token budget is dynamic: target 800-1200 tokens for context.

## Metadata Threading

After selecting a prompt, Conductor writes metadata to
`.prism/brain/.conductor_meta.json` for the stop hook to read
when recording the outcome:

```json
{
    "step_id": "write_failing_tests",
    "prompt_id": "dev/lean",
    "context_tokens": 847
}
```

This links the outcome record back to the exact prompt variant
and context size used, enabling accurate per-variant scoring.
